# Interactive FastAPI Demo Design

## Objective

Build an interactive local demo that turns the UAV Mission Intelligence Agent from a code-and-README proof into an operable project showcase. The demo should let a user enter a UAV mission, choose an execution provider, inspect the Agent trace and JSON output, compare benchmark/provider results, and view the generated mission situation map.

## Scope

- Add a FastAPI + HTML demo layer on top of the existing mission workflow.
- Support mission input, provider selection, Agent trace display, structured JSON output, benchmark/provider comparison, and mission situation visualization.
- Reuse the existing `run_agent_workflow`, LLM provider factory, Benchmark v2, scenario loader, schema validation, and mission SVG renderer.
- Keep the default project dependency-light. FastAPI and Uvicorn are optional demo dependencies, not required for normal CLI or tests.
- Keep the existing static dashboard available. The new demo is an interactive companion, not a replacement for the generated HTML dashboard.
- Document the demo setup and run commands in README and engineering notes.

## Non-Goals

- Do not replace the current Agent workflow, RAG layer, benchmark scoring, or provider adapter.
- Do not add a frontend build system, Node.js dependency, database, login system, or persistent server state.
- Do not call live providers unless the user explicitly selects a live provider and has configured the required API key.
- Do not require FastAPI or Uvicorn for the default unit test suite.
- Do not expose API key values in UI, logs, JSON responses, or error messages.

## Selected Approach

Use FastAPI + server-rendered HTML with a small amount of plain JavaScript. This gives the project a real service boundary and JSON API while avoiding frontend tooling overhead.

Alternatives considered:

- Streamlit: fastest to build, but it would make the project feel more like an experiment notebook than an application demo.
- Enhanced static HTML: keeps dependencies at zero, but cannot run new mission inputs or provider choices without a backend.

FastAPI is the best fit because the project already has clean Python workflow functions and JSON outputs. A thin API layer can expose those capabilities without changing core behavior.

## Architecture

Add three small demo-oriented units:

- `demo_service.py`: dependency-free orchestration helpers that build mission demo payloads, load benchmark data, and normalize errors.
- `demo_app.py`: optional FastAPI app factory and route handlers.
- `demo_cli.py`: small command-line launcher for the demo server.

`demo_service.py` is intentionally independent of FastAPI so its behavior can be tested in the default offline suite. `demo_app.py` imports FastAPI only inside the optional app boundary and raises a clear install hint when demo dependencies are missing.

The demo server exposes:

- `GET /`: returns the interactive HTML application.
- `POST /api/mission`: accepts mission text, provider, model, and base URL; returns the mission plan, trace, schema validation, LLM metadata, and mission SVG.
- `GET /api/benchmark`: returns provider comparison data, preferring the existing 31-scenario live comparison report when present and falling back to local offline Benchmark v2.
- `GET /api/health`: returns a small readiness payload for smoke tests.

## User Interface

The first screen is the working demo, not a landing page. The layout is a quiet mission-console style:

- Left control rail: mission text input, provider select, optional model/base URL fields, and run button.
- Main workspace: mission status summary, Agent trace timeline, JSON output, provider comparison table, benchmark summary, and mission situation map.
- Responsive behavior: desktop uses a two-column operational layout; mobile stacks controls above results.

The UI should stay compact and utilitarian. It should avoid marketing hero sections, nested cards, decorative backgrounds, and large explanatory copy. Controls should be stable in size so result updates do not shift the layout unexpectedly.

## Provider Behavior

Supported provider choices:

- `offline`: default, no API key required.
- `deepseek`: uses `DEEPSEEK_API_KEY`, optional `DEEPSEEK_MODEL`, and optional `DEEPSEEK_BASE_URL`.
- `openai-compatible`: uses `OPENAI_API_KEY`, optional `OPENAI_MODEL`, and optional `OPENAI_BASE_URL`.

The API should also accept a model override and base URL override for advanced local testing. The response includes provider and model metadata when a live provider is used. Missing API keys produce clear user-facing errors such as `missing API key: set DEEPSEEK_API_KEY or pass api_key`, with no secret values.

## Benchmark Flow

The benchmark panel should prefer project evidence already generated by Task 1:

1. Try to load `results/deepseek_provider_comparison_2026-07-09.json`.
2. If the file is absent or invalid, run local Benchmark v2 over `data/scenarios` with the offline provider.
3. Return normalized summary and provider comparison rows for the UI.

This keeps the demo fast and credible: it can show the real 31-scenario DeepSeek comparison when available, while still working offline from a fresh checkout.

## Data Flow

Mission run:

1. Browser sends mission request JSON to `/api/mission`.
2. Service validates non-empty mission text and provider name.
3. Service builds an optional LLM provider through the existing factory.
4. Service runs the existing traceable `run_agent_workflow`.
5. Service renders the mission SVG through `render_mission_execution_svg`.
6. Service returns a single payload containing status, plan JSON, trace, review, validation, LLM metadata, and SVG.

Benchmark load:

1. Browser requests `/api/benchmark`.
2. Service loads the saved 31-scenario report if possible.
3. Service falls back to offline Benchmark v2 if needed.
4. Browser renders summary metrics and provider comparison rows.

## Error Handling

- Empty mission text returns HTTP 400 with a short validation error.
- Unsupported provider names return HTTP 400.
- Missing optional FastAPI dependencies produce an install hint: `pip install 'uav-mission-intelligence-agent[demo]'`.
- Missing API keys and provider request failures return structured errors without secret values.
- Invalid benchmark artifact content triggers offline fallback rather than breaking the page.
- Unexpected server errors return a generic UI message plus a safe diagnostic string.

## Packaging And Commands

`pyproject.toml` will add:

```toml
[project.optional-dependencies]
demo = [
  "fastapi>=0.100",
  "uvicorn>=0.23",
]

[project.scripts]
uav-mission-agent-demo = "uav_mission_agent.demo_cli:main"
```

Run examples:

```powershell
pip install -e ".[demo]"
uav-mission-agent-demo --host 127.0.0.1 --port 8000
uav-mission-agent-demo --env-file D:\epacode\working\.secrets\deepseek.env
```

The launcher can load simple `.env` files containing `KEY=value` lines so the existing DeepSeek key location can be used without printing or committing secrets.

## Testing Strategy

Default offline tests:

- `demo_service` builds a complete mission payload with offline provider.
- Mission payload contains Agent trace, schema validation, JSON plan, and mission SVG.
- Empty mission text and unsupported provider names fail clearly.
- Saved benchmark report loading returns provider comparison rows.
- Invalid or missing benchmark report falls back to offline Benchmark v2.
- Demo HTML contains the mission input, provider selector, trace, JSON, benchmark, and mission map regions.
- `demo_cli` parses host, port, and env-file arguments without starting a server in unit tests.

Optional demo tests, only when FastAPI is installed:

- `GET /api/health` returns ok.
- `GET /` returns HTML.
- `POST /api/mission` returns an offline mission payload.
- Provider errors are returned as structured API errors.

## Acceptance Criteria

- The demo can be started locally with `uav-mission-agent-demo --host 127.0.0.1 --port 8000`.
- The page accepts a UAV mission and returns an Agent trace, structured JSON output, provider metadata, benchmark/provider comparison, and mission situation SVG.
- Offline mode works without API keys, network access, FastAPI in the default test environment, or live provider calls.
- DeepSeek mode uses configured environment variables and reports missing-key errors safely.
- The benchmark panel shows the saved 31-scenario provider comparison when available.
- `python -B -m unittest discover -s tests -v` passes.
- README documents how to install and run the demo.
